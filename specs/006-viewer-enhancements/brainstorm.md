# Brainstorm: Conversation Viewer Enhancements

**Status**: Ready for specification
**Captured**: 2026-03-08

## Idea

Improve the conversation viewer (detail pane) with richer markdown rendering, syntax highlighting, better visual separation between roles, and adaptive color themes.

## Problems with Current Viewer

### 1. System/synthetic content leaking through
The `strip_tags` function removes XML tag markers (`<...>`, `</...>`) but preserves the text content between them. This causes:
- "Caveat: The messages below were generated by the user..." text showing as user content
- Tool output (directory listings, file contents, bash results) displayed as if the user typed it
- The actual user message gets buried under pages of tool output

**Root cause**: `strip_tags` in `session.rs` only strips the tag delimiters, not the content between matching open/close tags. System-injected blocks like `<local-command-caveat>...</local-command-caveat>`, `<local-command-stdout>...</local-command-stdout>`, and `<command-message>...</command-message>` need their entire content stripped.

### 2. No syntax highlighting in code blocks
Code blocks use a single monochrome style (`Rgb(130,170,200)`). No language detection, no keyword/string/comment differentiation. Makes code blocks hard to scan.

### 3. Raw markdown tables
Tables render as raw pipe-separated text. No column alignment, no grid characters, no header styling.

### 4. No role visual differentiation
"▶ You" and "◀ Claude" are small colored text spans. No full-width background, no message body differentiation. Hard to visually scan where one role's messages end and another's begin.

### 5. Hardcoded dark theme only
All colors are hardcoded RGB values optimized for dark terminals. Light terminal users get poor contrast. No detection, no override.

## Proposed Enhancements

### Enhancement 1: Strip system-injected content blocks

Instead of just removing XML tag markers, strip entire content blocks for known system tags:
- `<system-reminder>...</system-reminder>` (already fixed in search, need to fix in viewer)
- `<local-command-caveat>...</local-command-caveat>`
- `<local-command-stdout>...</local-command-stdout>`
- `<command-name>...</command-name>`
- `<command-message>...</command-message>`
- `<command-args>...</command-args>`
- `<user-prompt-submit-hook>...</user-prompt-submit-hook>`
- `<available-deferred-tools>...</available-deferred-tools>`
- `<rosa-auth>...</rosa-auth>`, `<rosa-profile>...</rosa-profile>`
- `<sdd-context>...</sdd-context>`
- `<skill-enforcement>...</skill-enforcement>`
- `<fast_mode_info>...</fast_mode_info>`

**Approach**: Replace `strip_tags` (which strips ALL tags) with a smarter function that:
1. Strips entire content blocks for known system tags (tag + content)
2. Strips only the tag markers for other tags (preserve user-visible HTML-like content)
3. Filters out entries that become empty after stripping (like pure-caveat messages)

### Enhancement 2: Syntax highlighting with language detection

Use `syntect` (Sublime Text syntax definitions, used by `bat`) for full syntax highlighting:
- Parse the language hint from the code fence opening (e.g., ` ```rust `, ` ```python `)
- Apply syntax theme colors to highlighted tokens
- Fall back to the current single-color style if language not recognized
- Theme selection tied to light/dark mode detection

**Why syntect over tree-sitter**: syntect is a pure-Rust library with built-in syntax definitions and themes. It's lighter weight than tree-sitter (which requires per-language parser binaries) and integrates naturally with ratatui's `Span`-based rendering.

**Dependencies**: `syntect` (~2MB added to binary for syntax definitions)

### Enhancement 3: Markdown table rendering

Full grid rendering with Unicode box-drawing characters:
```
┌──────────┬─────────┬───────────┐
│ Column 1 │ Column 2│ Column 3  │
├──────────┼─────────┼───────────┤
│ data     │ data    │ data      │
│ more     │ more    │ more      │
└──────────┴─────────┴───────────┘
```

**Implementation approach**:
1. Detect table blocks (consecutive lines starting with `|`)
2. Parse cells and compute max column widths
3. Render with box-drawing characters: `┌─┬─┐`, `├─┼─┤`, `└─┴─┘`, `│`
4. Header row rendered bold
5. Handle word wrap within cells if table exceeds content width
6. Skip the markdown separator row (`|---|---|`)

### Enhancement 4: Role header bars and message backgrounds

**Full-width role header bars**:
- "▶ You" rendered as full-width bar with colored background (e.g., Cyan bg, dark text)
- "◀ Claude" rendered as full-width bar with different colored background (e.g., Yellow bg, dark text)
- Timestamp right-aligned within the bar (replacing the separate separator line)

**User message background tint**:
- User message body lines get a subtle background tint (e.g., `Rgb(25,30,40)` on dark, `Rgb(235,240,245)` on light)
- Assistant message body keeps default terminal background
- Creates visual "chat bubble" effect without borders

### Enhancement 5: Light/dark theme detection and adaptation

**Auto-detection**:
1. Check `COLORFGBG` environment variable (set by many terminals, format: "fg;bg" where bg >= 8 suggests dark)
2. Try OSC 11 terminal query (sends escape sequence, reads terminal background color response)
3. Default to dark theme if detection fails

**Manual override**: `--light` / `--dark` CLI flags, plus `CC_SESSION_THEME` env var

**Theme structure**: Define a `Theme` struct with named colors:
- `role_user_bg`, `role_user_fg`
- `role_assistant_bg`, `role_assistant_fg`
- `user_message_bg`
- `code_block_bg`
- `heading_fg`
- `separator_fg`
- `search_highlight_bg`

Two built-in themes (dark, light) with appropriate colors for each.

## Dependencies

- `syntect` for syntax highlighting (new dependency)
- No other new dependencies needed
- Existing `crossterm` supports OSC queries for terminal detection

## Complexity Estimate

- **Enhancement 1** (strip system content): Low complexity. Modify `strip_tags` or add `strip_system_blocks`. Primarily string processing.
- **Enhancement 2** (syntax highlighting): Medium complexity. Integrate syntect, map highlighted tokens to ratatui Spans, handle theme selection.
- **Enhancement 3** (table rendering): Medium complexity. Table parsing, column width calculation, box-drawing character rendering, word wrap within cells.
- **Enhancement 4** (role headers/backgrounds): Low complexity. Change line styles in `pre_render_conversation`, add background to Line styles.
- **Enhancement 5** (theme detection): Medium complexity. OSC 11 query is async/timeout-based, theme struct definition, propagation through rendering code.

## Suggested Implementation Order

1. Enhancement 1 (strip system content) - immediate UX improvement, low risk
2. Enhancement 4 (role headers/backgrounds) - high visual impact, low complexity
3. Enhancement 5 (theme detection) - needed before syntax highlighting themes
4. Enhancement 3 (table rendering) - self-contained, medium complexity
5. Enhancement 2 (syntax highlighting) - largest change, depends on theme system

## Open Questions

- Should tool use entries be shown at all? Or should they be collapsed/summarized (e.g., "Used 3 tools" with expand option)?
- Should the viewer support collapsible sections for long code blocks?
- Should we add line numbers to code blocks?
- How should markdown lists (bullet, numbered) be rendered? Currently they're plain text.
